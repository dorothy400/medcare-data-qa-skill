# 查询约束

## 查询规则

- 只能使用已批准的 SQL 模板。
- 只能查询已批准的数据集。
- 只能替换已批准的参数。
- 当表字段存储格式与通用时间参数格式不一致时，必须先做参数格式对齐后再下推到 SQL。
- 当答案中包含“数据更新时间”时，必须额外执行一条元数据查询。
- 排名类或明细类查询必须限制返回规模。

## biz_date 对齐规则

- 当前数据集中的 `biz_date` 字段存储格式为 `yyyy-MM`。
- 运行层应先根据时间规则求出 `start_date` / `end_date`，再派生出 `start_month` / `end_month`。
- 主查询与元数据查询都必须使用格式对齐后的 `start_month` / `end_month` 过滤 `biz_date`。
- 当用户问“本月 / 本季度 / 今年 / 本年”等当前周期时，应补全为完整自然周期，不按现实日期截断。
- 例如：如果当前日期是 2026-05-15，则“今年”应解析为 `2026-01 ~ 2026-12`，而不是 `2026-01 ~ 2026-05`。

## 允许的筛选与聚合维度

- 以下字段既可以作为筛选条件，也可以作为聚合维度：`nick_name`、`source`、`dept`、`director_nick_name`、`manager_nick_name`
- 当用户使用这些字段的中文业务称呼提问时，运行层应先映射到标准字段名，再拼装到 SQL 模板中。
- 如果用户要求“按某维度拆分/汇总/排名”，只能使用已批准的维度字段，不允许自由使用未登记字段做 `group by`。

## 数据更新时间查询规则

- 数据更新时间应通过 `max(update_time)` 获取。
- 元数据查询使用独立批准的数据集与模板，不要求与主查询使用同一数据集。
- 元数据查询返回字段统一命名为 `last_update_time`。
- 元数据查询按其自身模板要求解释参数，不要求与主查询筛选条件完全一致。

## 查询时必须记录的元数据

每次执行查询时，运行层都应记录这些标签：

- metric_name
- template_name
- requester
- request_time
- resolved_filters
- row_limit
- metadata_template_name

## 不支持的情况

- 未知指标
- 当前不支持的粒度
- 受保护模板缺少必填筛选条件
- 超出允许范围的原始明细提取
